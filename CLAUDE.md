## Project Overview

**Productivy** — a MERN-stack (React+Vite+Tailwind client, Express+MongoDB server) personal project/sprint/notes tracker. Built by Akram right after his final-year internship (PFE/WeStart), partly to learn the MERN stack, partly as a real tool: track projects/sprints/notes, a Pomodoro timer, and a "time worked" tracker based on an online-presence heartbeat. Trello was a design inspiration. Auth via Google OAuth (Passport).

Live at **productivy.vercel.app** (client). Server CORS whitelists that origin.

## Architecture

- `client/` — React + Vite + Tailwind. Key components: `ProductivityDashboard.jsx`, `DayActivityGraph.jsx`, `ProductShowcase.jsx`.
- `server/` — Express + MongoDB (Mongoose). Models: `Activity`, `CurrentActivity`, `ProductiveTime`, `Project`, `User`, `WeeklySummary`, `Average`.
- `server/cron/activityCron.js` — the heartbeat job, runs every minute, walks online users and increments their activity/time-worked buckets. See "Known Issues" below — this is the area most in need of a rework.

### Activity/time-worked logic — ✅ elapsed-time accounting fixed 2026-07-15, commit `c5938df`

`activityCron.js` used to increment each online user's time-worked bucket by a flat 1 minute per cron tick regardless of real elapsed time. Fixed: added `CurrentActivity.lastAccountedAt` and now credit `min(now - lastAccountedAt, 90s)` per tick — real elapsed time, capped so a missed/delayed cron run (server restart, cold start) can't dump a huge backlog once it resumes. First tick after coming online bootstraps and credits 0. `Activity.hours.*` and `ProductiveTime.minutes` now accumulate fractional minutes (both are plain Mongoose `Number`, floats are fine) instead of always whole numbers.

**Explicitly deferred, not fixed**: day/hour bucketing still uses server-local time (`now.getFullYear()/getMonth()/getDate()`), unchanged. This convention is used consistently in three places — `activityCron.js`, `activityController.js`'s `getTodayProductiveTime`, and `chartsController.js`'s daily/weekly day-string generation — so it's internally consistent, just potentially not matching the deployed server's actual timezone vs. the user's. Fixing this properly means touching all three together (to avoid a reads-vs-writes mismatch) and possibly adding a stored user-timezone preference — bigger scope than this pass, and not something to do solo without Akram's input on how much precision he actually wants here. Revisit as its own task.

Correctness still depends on the client-side `isOnline`/`lastSeen` heartbeat being robust (handling tab-close/disconnect cleanly) — not touched in this pass, no evidence it's currently broken.

Verification note: no test suite exists in this repo (`npm test` is a no-op placeholder). Verified via `node --check` on both changed files and a real local server boot (connects to the actual MongoDB via `.env`, no runtime errors) — not a full integration test of the cron's actual crediting behavior over time. Worth a manual sanity check (watch `ProductiveTime.minutes` over a few real minutes online) whenever this app is next actually used.

## Second bug/feedback pass (2026-07-22) — ✅ 6 items shipped

Akram reported 6 more issues after continued real use:

1. **Sprint double-click-to-rename reloaded the task list** — `Sprints.jsx`: a native double-click fires two ordinary `click` events before the `dblclick` event itself, and each of those bubbled to `handleSprintClick`, switching `selectedSprintId` (and thus which sprint's tasks `Kanban.jsx` shows) *before* rename mode even opened — visible as the task list reloading to a different sprint's tasks. Fixed with a ~220ms debounce: a single click now waits briefly before actually selecting, and `handleDoubleClick` cancels that pending selection.
2. **Long column titles overflowed instead of truncating** — `Column.jsx`'s `<h3>` already had `truncate`, but sat inside a flex row without `min-w-0`. Flex items default to `min-width: auto` (their content's natural width), which silently defeats `overflow:hidden`/ellipsis truncation in a flex context — a well-known Tailwind/flexbox gotcha. Added `min-w-0` to both the `<h3>` and its flex parent; also added a `title` attribute so the full name is still visible on hover.
3. **No practical input length limits anywhere** — added `maxLength` on every free-text input client-side (project/sprint/board/column/card titles, task notes/descriptions, username/email/password) *and* matching server-side enforcement: `maxlength` on the relevant `Project.js`/`User.js` Mongoose schema fields, plus explicit trim+length checks in `projectController.js` (a `MAX_LEN` constants object) and `authRoutes.js`'s register handler, so oversized input gets a clean 400 instead of relying on the client or a raw Mongoose `ValidationError`. Password capped at 72 — bcrypt silently truncates anything past 72 bytes, so a longer max would be actively misleading (two different passwords sharing the first 72 bytes would authenticate as the same one).
4. **"Time worked" showing raw floats (e.g. `23.000256`)** — traced to `DayActivityGraph.jsx`'s hover tooltip (`Minutes: {tooltipData.minutes}`), which displays the accumulated fractional-minute value straight from the DB with no rounding (unlike `Time.jsx`/`ProductivityDashboard.jsx`, which already round via `formatTime`). Resolved by item 5 below rather than patched directly, since the whole component is going away.
5. **Daily Activity Timeline commented out** — Akram decided it's not useful now that the dashboard has real stats. `DayActivityGraph` import + usage commented out in `Account.jsx` (same pattern as the already-commented `WeeklyActivityGraph` next to it), component file left in place, not deleted.
6. **Investigated "the whole site feels slow"** — the real, fixable culprit found: `rateLimiter` middleware (`server/middleware/rateLimiter.js`) called Upstash Redis on *every single request*, GET included, using one global rate-limit key — meaning every read (the dashboard alone fires ~9 parallel GETs on load) paid a full extra network round-trip to Redis before even reaching Mongo. Changed it to skip GET/HEAD entirely and only rate-limit mutating requests. Also added an index on `Project.userId` (queried on every `/api/projects` fetch, previously unindexed). **Not code-fixable, flagged instead**: Render's free tier spins the server down after ~15 min idle (30-60s cold-start on the next request), and Mongo Atlas's free M0 tier is shared/throttled and may be in a different region than Render — both are inherent to the free-tier hosting choice, not bugs.

**Verification**: server boots clean (`mongoose connected`) and `npm run build` passes with all changes in place. The sprint-rename and column-truncation fixes were visually confirmed via the temp-bypass technique (see note above) — stubbed `Project.jsx`'s and `Board.jsx`'s fetch `catch` blocks with fake data including a long column title and a second empty sprint, confirmed via computed styles (`scrollWidth` > `clientWidth`, single-line height) that truncation works, and confirmed via DOM query that double-clicking a non-selected sprint enters rename mode without the Sprint Tasks panel switching away. Both stubs fully reverted before finishing — `git diff` on `Project.jsx` is empty, `Board.jsx`'s diff shows only the intended `maxLength` addition.

**Follow-up same day**: Akram asked to sync the drift and keep dev servers running while he tested live. Synced both directions — committed the dangling `upstash.js` rate-limit tightening in the main repo, then caught `Productivy-api` up on everything already shipped in the main repo (Focus module backend, Event model, activity-status collapse, the rate-limiter/index fixes above).

**Hotfix, same day**: adding `maxlength` directly on the `Project.js`/`User.js` Mongoose schemas (see item 3 above) turned out to be a real regression, caught live via server logs while Akram was testing — `changeColumnOrder` started 500ing on totally unrelated column-reorder actions. Root cause: nearly every mutation in this app calls `.save()` on the *whole* embedded Project/User document, and Mongoose validates the entire tree on every save, not just the changed path — so pre-existing data that predated the new limits (a 44-char column title, a 185-char card title) broke unrelated saves outright. Reverted the schema-level `maxlength` in both repos; the length limits are still fully enforced via the `projectController.js`/`authRoutes.js` checks, which only validate the specific field being written. **Lesson for next time**: don't add Mongoose-schema-level `maxlength`/similar validators to documents that get saved as a whole embedded tree unless you're sure no existing data violates them (or use `validateModifiedOnly`) — controller-level checks on the specific field are the safer default here.
